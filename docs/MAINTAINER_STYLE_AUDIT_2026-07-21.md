## 2. Maintainer Simulation Audit

### What "simulation" means in code

`SimulationEngineImpl` (`kri/simulation/engine.py:67`) means **reproducible pipeline execution
against a snapshotted knowledge state** — not impersonation of a specific maintainer. Steps:
snapshot EKG state → load DKP → call `ReviewEngine.review()` → call
`ReportGenerator.generate()` → attach metadata. The `replay()` method (`engine.py:123`) restores
a prior snapshot for byte-identical re-runs. The `audit()` method (`engine.py:148`) SHA-256
hashes the canonical JSON for traceability.

### Does KRI simulate reviewer decisions / tone / style / history?

| Dimension | Status | Evidence |
|---|---|---|
| Reviewer decisions | **Yes — rule-based only** | `PatternMatchPlugin.evaluate()` (`plugins.py:183`) returns pass/fail per pattern; `ConfidenceEngine` scores it. LLM agents return severity-classified findings. Neither uses a named maintainer's decision record. |
| Reviewer tone | **Generic only** | `SYSTEM_KERNEL_REVIEWER` (`prompts.py:5`): *"You are an experienced Linux kernel maintainer… direct but constructive, like real kernel reviewers on lore.kernel.org."* No named individual, no voice data. |
| Reviewer communication style | **Generic only** | The `upstream_comment` field instruction adds: *"Sound like a subsystem maintainer asking a pointed question, not a static analysis tool."* One synthetic illustrative example included. No real lore email quotes. |
| Reviewer history | **Not implemented** | `REVIEW_HISTORY` confidence factor (`confidence_engine/engine.py:192–200`) counts verified `REVIEW_DISCUSSION` evidence nodes. These exist only if the DKP pre-seeded them (3 rules have lore-backed evidence). No dynamic history lookup at review time. |

### Is the lore reply generated from maintainer examples, prompt instructions, or generic LLM reasoning?

The system produces two separate "lore-style" outputs:

**A. `format_lore_reply()` (`kri/llm/formatter.py:9`):**
A deterministic Python string formatter — no LLM call. It quotes diff context lines with `>`,
writes `[BLOCKER]`/`[WARNING]` tags, appends `comment.message` (not `upstream_comment`), and
closes with a KRI disclaimer. This is a **synthetic format** — not modeled on any real lore email
structure. It does not use the `Reviewed-by:` / `Nacked-by:` / `In-Reply-To:` conventions of
actual lore threads.

**B. `InlineComment.upstream_comment` (per-finding):**
Generated by the LLM via the `REVIEW_CODE_QUALITY_PROMPT`/`REVIEW_SUBSYSTEM_PROMPT` templates.
This is **prompt instruction only** (type A from the taxonomy below). No real lore comment
examples are in the prompt.

**Verdict: Type A — prompt instruction only.** KRI does not use few-shot examples of real lore
comments, is not fine-tuned, and does not retrieve similar historical comments (RAG).

## 3. Lore Comment Corpus Audit

### What is stored

**`data/lore_cache/` — 129 `.mbox.gz` files** (real patch submission threads from lore.kernel.org,
ASoC / alsa-devel, 2022–2026). Each is a full public-inbox thread export containing both patch
emails **and** maintainer reply threads including all inline review comments.

Three files are designated canonical ground-truth fixtures (`data/lore_cache/FIXTURES.json`):

| Fixture | Content |
|---|---|
| `20260630021510_821919-3-YLCHANG2_nuvoton_com.mbox.gz` | Nuvoton NAU83G60 v5 — 9 messages, Mark Brown + Krzysztof Kozlowski reviews |
| `20260708093506_895481-1-YLCHANG2_nuvoton_com.mbox.gz` | NAU83G60 v6 respin — 5 messages |
| `20260703123314_147977-1-syed_sabakareem_amd_com.mbox.gz` | AMD clean-path (no review comments) |

The remaining 126 files are the `LoreManager`'s on-disk fetch cache — real threads but not
formally indexed as fixtures.

### How the corpus is used

| Component | How corpus is used | Wired to live review output? |
|---|---|---|
| `lore_manager/manager.py:225` `extract_reviews()` | Parses mbox threads into `ReviewComment` objects with `is_maintainer` flag | Used for benchmark and tests; **not** fed into LLM prompts at review time |
| `benchmark/runner.py:81` | Takes ground-truth `ReviewComment` list from lore fixtures, compares against KRI decisions | Wired to `tests/test_benchmark_regression.py`; not part of the live `/api/review` flow |
| `learning/extraction.py:149` `HistoricalPatternExtractor` | Classifies maintainer concern categories across threads; emits `CandidatePattern` dicts | **Not wired** to review flow. Reports "insufficient" support for all patterns (< 5 examples threshold). |
| `packages/asoc/knowledge.py:142–184` | Three specific Mark Brown lore message-IDs cited as rule provenance | Wired — creates `REVIEW_DISCUSSION` evidence nodes that feed the `REVIEW_HISTORY` confidence factor (weight 0.10) |

**Are review comments from the corpus currently influencing review output?** Indirectly and
minimally. The three lore message-IDs seeded in the ASoC DKP contribute to the `REVIEW_HISTORY`
confidence factor in rule-based reviews only. For the LLM path, the corpus has zero influence on
what the LLM writes — none of the 129 mbox files are read at review time.

## 4. Reviewer Style Learning Feasibility

### Named reviewers in the codebase

| Reviewer | Present in code | Role |
|---|---|---|
| Mark Brown | Yes — `knowledge.py:39`, lore citations at lines 142–184 | Identity node + 3 rule provenance citations |
| Liam Girdwood | Yes — `knowledge.py:41` | Identity node only |
| Bartosz Golaszewski | Not present | — |
| Takashi Iwai | Not present | — |
| Jaroslav Kysela | Not present | — |
| Krzysztof Kozlowski | Comments and tests only | NAU83G60 fixture annotation |

### Infrastructure that already supports style learning

1. **`HistoricalPatternExtractor` (`learning/extraction.py:149`)** — reads real lore threads,
   classifies maintainer concerns by category using `_CONCERN_SIGNALS` lexicon (`line 57`), filters
   to maintainer-only change-requests, groups into `CandidatePattern` objects. This is
   **structural** pattern learning (api_misuse, error_handling, design, style, etc.) — not
   voice/phrasing learning.

2. **`_CONCERN_SIGNALS` lexicon (`extraction.py:57`)** — already categorizes concerns under:
   `nack`, `approval`, `error_handling`, `locking`, `api_misuse`, `design`, `documentation`,
   `style`. A maintainer-style-aware prompt could use these category labels as tone anchors.

3. **`extract_reviews()` + `ReviewComment.is_maintainer`** — the system already knows which reply
   emails are from maintainers vs. submitters. Extracting verbatim maintainer comment text for
   few-shot injection is mechanically straightforward.

4. **EKG Maintainer nodes** — `NODE_MAINTAINER` exists in the schema (`schema.py`), Mark Brown
   and Liam Girdwood are already seeded. The `EDGE_REVIEWS` edge type is defined (`schema.py:58`)
   but never created. Wiring it would enable "which maintainer reviews this subsystem" lookups.

### Extractable style patterns (subsystem-community voice, not person-specific)

The corpus (especially the NAU83G60 threads) contains examples of these patterns that can be
extracted without attributing them to any individual:

| Pattern | Extraction approach |
|---|---|
| Asking clarifying questions | `_CHANGE_REQUEST_RE` classifies these; extract full sentence |
| Avoiding hard assertions | Lexical: "seems like", "could be", "I'd expect", "have you considered" |
| Requesting justification | Lexical: "why", "please explain", "what's the rationale" |
| Pointing out inconsistencies | Category: `design`, `api_misuse` |
| Short approvals ("Looks good to me") | Already classified as `approval` |
| Style nitpicks | Category: `style`; tied to checkpatch/whitespace signals |

### What a few-shot prompt would look like

Instead of the current generic persona (*"You are an experienced Linux kernel maintainer…"*), a
few-shot approach injects 2–3 actual lore review comment excerpts as examples:

```
Here are examples of how ASoC maintainers phrase review concerns:

Example 1 (resource cleanup):
> kzalloc(sizeof(*priv), GFP_KERNEL)
^
This will leak if the subsequent initialization fails. Please add an
error-unwind path, or use devm_kzalloc and ensure all resources use
devm_ variants consistently.

Example 2 (API choice):
Should use snd_soc_component_read() here rather than reading the
regmap directly — the component API handles the power state correctly.
```

This requires extracting verbatim maintainer text from the mbox corpus (infrastructure exists) and
injecting it into the prompt at review time. No fine-tuning needed.

## 5. Gap Analysis

### A — Already implemented and wired

- Lore corpus download and mbox parsing (`lore_manager/manager.py`)
- `ReviewComment.is_maintainer` classification from MAINTAINERS file (`maintainers.py`)
- EKG Maintainer identity nodes for Mark Brown and Liam Girdwood (`plugin.py:161–169`)
- Lore message-ID provenance citations for 3 ASoC rules (`knowledge.py:142–184`)
- `REVIEW_DISCUSSION` evidence type contributing to `REVIEW_HISTORY` confidence factor
- `upstream_comment` field populated by LLM agents and rendered in UI
- Generic maintainer persona in `SYSTEM_KERNEL_REVIEWER`

### B — Implemented but unused / disconnected

- **`format_lore_reply()` does NOT use `upstream_comment`** (`formatter.py:64`): uses
  `comment.message` instead. The maintainer-voiced text the LLM was explicitly prompted to write
  is silently dropped in the lore email output.
- **`HistoricalPatternExtractor` (`learning/extraction.py:149`)**: complete extraction pipeline,
  but never called from any review endpoint.
- **`EKG EDGE_REVIEWS` edge type**: defined in `schema.py:58`, never seeded.
- **126 mbox files beyond the 3 canonical fixtures**: in the cache, never ingested for learning.
- **`render_maintainer_comment()` (`report/maintainer_template.py:27`)**: formats rule-based
  decisions in lore quote style (`> offending code`, concern, suggested fix, precedent links),
  but NOT called from the web API response path — only used in tests.

### C — Partially implemented

- **Learning pipeline**: `HistoricalPatternExtractor` extracts concern categories but outputs
  "insufficient" support for all patterns (only 3 canonical fixtures; ≥5 threshold requires more).
- **Precedent mechanism**: works for the 3 ASoC rules that have it; the 15+ other rules have no
  precedents (`_NO_PRECEDENT_PENALTY` = 0.90× applied).
- **`hunk_context` field on `InlineComment`**: defined, LLM inconsistently fills it, not reliably
  extracted from diff in the backend.

### D — Architecture planned, absent in implementation

- Per-maintainer voice simulation: `ASOC_MAINTAINERS` data structure exists, `EDGE_REVIEWS`
  schema defined, but no mechanism routes from "this patch targets sound/soc/" → "use phrasing
  patterns derived from Mark Brown's review history."
- RAG over lore corpus: no vector store, no embedding pipeline, no similarity retrieval at review
  time.
- Subsystem-aware few-shot prompt injection: `build_domain_context()` (`prompts.py:190`) formats
  rule labels but has no path to inject actual comment examples.

### E — Completely new work

- Few-shot example extraction from mbox corpus into prompt template
- `format_lore_reply()` using `upstream_comment` instead of `comment.message`
- `hunk_context` extraction from diff in `_review_patch()` (Phase 2 item from ARCHITECTURE_AUDIT)
- Temperature increase for `upstream_comment` generation to capture phrasing variation
- Multi-maintainer style profiles (out of scope until corpus is large enough)

## 6. Recommended Next Steps

Prioritized by impact to review quality against current state:

| Priority | Change | Gap class | Effort | Impact |
|---|---|---|---|---|
| 1 | Wire `upstream_comment` into `format_lore_reply()` | B — broken wire | Trivial (5 lines) | Immediate lore reply quality jump |
| 2 | Few-shot real lore examples in prompts | D/E — new | Low | Biggest tone shift |
| 3 | Extract `hunk_context` from diff in reviewer | C — partial | Medium | Anchors comments to code |
| 4 | Wire `HistoricalPatternExtractor` to expand DKP | B — unused | Medium | More rule coverage |
| 5 | Patched-tree validation (repo_manager + checkpatch) | B — not wired | High | Correctness, not style |

### Priority 1 detail — fix `format_lore_reply()`

In `kri/llm/formatter.py:64`, replace `comment.message` with `comment.upstream_comment or
comment.message`. The LLM already generates high-quality maintainer-voiced text for every finding;
the formatter silently discards it.

### Priority 2 detail — few-shot injection

Extract 3–5 verbatim maintainer comment excerpts from the NAU83G60 corpus (Mark Brown's review
replies in `data/lore_cache/20260630021510_821919-3-YLCHANG2_nuvoton_com.mbox.gz`), inject them
into `SYSTEM_KERNEL_REVIEWER` or the `upstream_comment` field description. The concern categories
(`error_handling`, `api_misuse`, `design`) from `_CONCERN_SIGNALS` can be used to select the
most relevant example per finding type. No new infrastructure required.

### Priority 3 detail — `hunk_context` extraction

`format_lore_reply()` already has `_find_context()` (`formatter.py:50–60`) that extracts diff
window for a given line number. The same logic should be called in `reviewer.py:_review_patch()`
after agents return, back-filling `InlineComment.hunk_context` deterministically.
