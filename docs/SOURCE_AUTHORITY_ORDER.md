# Source Authority Order

**Date:** 2026-07-24
**Status:** Active — governs all autonomous execution decisions
**Scope:** Defines the precedence procedure for resolving conflicts between
architecture documents, implementation artifacts, and source evidence.

---

## 1. Purpose

This document does not define a linear ranking of sources. It defines a
**conflict-resolution procedure** — a decision tree applied whenever two
sources make contradictory claims about KRI's behavior, design, or intent.

The distinction matters: a linear ranking would silently resolve every conflict
by discarding one source. This procedure instead requires explicit recording of
every conflict before any resolution is applied.

## 3. Conflict Resolution Procedure

### Step 1 — Classify the conflict

| Type | Description |
|------|-------------|
| `INTENT_VS_REALITY` | Blueprint claims capability X; code implements Y differently |
| `VERSION_CONFLICT` | Two Layer-1 or two Layer-3 documents disagree |
| `SCOPE_CONFLICT` | General rule vs specific rule covering a narrower case |
| `PROVENANCE_CONFLICT` | Human-authored source vs LLM-generated source disagree |

### Step 2 — Apply resolution by type

**INTENT_VS_REALITY:**
Neither source wins automatically. This is a **reconciliation finding**. Record in
`BLUEPRINT_RECONCILIATION.md` with:
- Blueprint claim + citation
- Code behavior + file:line
- Drift category: `intentional_evolution` / `incidental_regression` / `unresolved_gap` / `blueprint_aspirational`
Do not resolve silently. Do not implement the blueprint blindly if the implementation
represents a deliberate engineering improvement. Do not preserve the implementation
blindly if it represents an accidental regression.

**VERSION_CONFLICT:**
1. Is the newer document explicitly superseding the older? Check for "supersedes",
   "replaces", or a direct citation of the older document name in the newer one.
2. If yes: newer wins within its stated scope.
3. If no: treat as INTENT_VS_REALITY (the conflict is unresolved) and record it.

**SCOPE_CONFLICT:**
Specific wins within its scope. General applies outside that scope. Record the
boundary definition as an assumption if it is not explicit in either document.

**PROVENANCE_CONFLICT:**
Layer-1 always wins over Layer-2. Record that the RTF's content was overridden and
why.

### Step 3 — Default precedence (fallback when Steps 1–2 don't apply)

When no conflict exists and multiple sources are informative:

```
L1-A (Constitution)
  > L1-B (ASoC model) / L1-C (Packages) / L1-D (Skills)   [equal peers]
  > L1-E (top-level product doc)
  > L3 in-repo specs (chronologically; newer supersedes if explicitly cited)
  > L4 module docstrings + commit messages
  > L2-A (RTF — LLM output)
  > L4 current code (what IS, never what SHOULD BE)
```

### Step 4 — Unresolvable conflicts

If a conflict does not fit any of Steps 1–3, it must be recorded in
`BLUEPRINT_RECONCILIATION.md § Conflicts` with status `UNRESOLVED`.
Autonomous execution does not proceed on any task whose scope touches an
UNRESOLVED conflict until the conflict is either:
- Classified as `blueprint_aspirational` (code wins)
- Classified as `intentional_evolution` (code wins, blueprint should be updated)
- Explicitly adjudicated by the user

---

## 4. Commit Message Authority Rule

Commit messages are **supporting evidence only**. They explain why an
implementation decision was made, which is valuable for classifying drift as
`intentional_evolution` vs `incidental_regression`. They never override blueprint
intent. They never justify removing a blueprint-required capability.

all conflict resolution purposes. Its content may be used to fill gaps in Layer-1
coverage, but it does not override any explicit Layer-1 claim.

**Assumption recorded:** this classification is based on filename evidence. If the
user confirms or disputes the RTF's authorship, this file must be updated.
